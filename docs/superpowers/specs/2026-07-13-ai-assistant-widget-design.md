# مساعد التقارير المدمج (AI Assistant Widget) — Design Spec

**Date:** 2026-07-13
**Status:** Approved

## الهدف

خانة محادثة عائمة داخل النظام يكتب فيها المستخدم طلبه بالعربي، فيفهمه النظام ويجلب التقرير المناسب من البيانات الموجودة — **بدون أي نموذج ذكاء اصطناعي خارجي، بلا مفتاح API، بلا تكلفة، ويبقى النظام offline**. يعتمد على مُحلّل نصّي مدمج يحوّل الطلب إلى "قصد منظّم" ثم يشغّل استعلامات التقارير الآمنة الموجودة أصلًا.

## القرارات المعتمدة

- **المزوّد:** مُحلّل نصّي مدمج (لا Ollama، لا سحابة). النظام يبقى offline بلا اعتماد جديد.
- **المخرجات:** يعرض النتيجة داخل الشات (جدول + مؤشرات) + ملخّص لغوي عربي + رابط تصدير.
- **الواجهة:** ودجت عائم يظهر في كل الصفحات.
- **التغطية:** التقارير الخمسة كلها — المبيعات، العملاء، أعمار الذمم، المخزون، المالية.

## البنية والمكوّنات

المبدأ الحاكم: **لا استعلامات SQL جديدة ولا نماذج خارجية.** يُعاد استخدام دوال الـ Repositories الموجودة والمختبَرة.

| المكوّن | الموقع | المسؤولية |
|--------|--------|-----------|
| `Intent` (نواة نقية) | `app/Core/Intent.php` | يحوّل النص العربي → مصفوفة `{report, from, to, search, filters, topN, export, format}`. لا يلمس قاعدة البيانات. قابل للاختبار وحده مثل `Report`/`Analytics`. |
| `AssistantController` | `app/Controllers/AssistantController.php` | يستقبل الطلب، يستدعي `Intent::parse()`، يشغّل دالة التقرير المناسبة، يبني رد JSON. |
| مسار `/assistant/ask` | يُسجّل في `public/index.php` | نقطة الاستدعاء عبر `fetch` (نفس النطاق ← يمرّ عبر `connect-src 'self'`). |
| الودجت (JS + CSS) | `assets/js/app.js` + `assets/css/app.css` | زر عائم + نافذة محادثة، يُضمَّن في `layout.php` فيظهر بكل الصفحات. |

**تدفّق الطلب:** إدخال المستخدم → `fetch('/assistant/ask?q=...')` → `Intent::parse()` → دالة Repository المناسبة → `Controller::json()` → عرض في الودجت.

## نطاق `Intent`

**نوع التقرير** (كلمات مفتاحية):
- `sales`: مبيعات، فواتير، بيع
- `customers`: عملاء، عميل، زبائن، زبون
- `aging`: ذمم، مستحقات، متأخرات، أعمار
- `inventory`: مخزون، أصناف، مخزن، نافد، حد الطلب، منتجات
- `finance`: مالية، مصروفات، مصاريف، إيرادات، تدفق، صافي

**الفترة الزمنية** (نسبيًا لأحدث تاريخ في بيانات المبيعات — `dateBounds()['max']`):
- "آخر N يوم/يومًا/أيام"
- "آخر N شهر/أشهر" (N شهرًا)
- "آخر سنة / آخر N سنوات" → 12 شهرًا (أو N×12)
- "هذا الشهر" / "الشهر الحالي" · "الشهر الماضي/السابق"
- "هذا العام / هذه السنة" · "العام الماضي"
- سنة صريحة: "٢٠٢٦" أو "2026" → كامل السنة
- شهر + سنة: "يناير ٢٠٢٦" → كامل الشهر
- يدعم الأرقام العربية (٠-٩) والإنجليزية (تُطبَّع قبل التحليل).

**مرشّحات:**
- اسم عميل: "عميل X" / "لعميل X" / "للعميل X" → `search = X`
- حالة المخزون: "نافد/نفد" → `status=out` · "منخفض/تحت الطلب/حد الطلب" → `status=low`
- نوع مالي: "مصروفات/مصاريف" → `type=expense` · "إيرادات" → `type=income`
- "أعلى/أكثر N" → `topN` (منتجات في المبيعات، أو عملاء)
- نيّة التصدير: "صدّر/تصدير/CSV/إكسل/اكسل/excel" → `export=true` مع `format` (csv افتراضيًا، xls لو ذُكر إكسل)

**الافتراضات:** لو لم تُذكر فترة → بلا فلترة زمنية (كل البيانات). لو تعذّر تحديد نوع التقرير → قصد فارغ (`report=null`) يؤدي لرسالة "لم أفهم".

## الشفافية والغموض

- **الشفافية:** كل رد ناجح يبدأ بسطر `understood` يوضّح التفسير: *"تقرير المبيعات · من ٢٠٢٦-٠٤-١٤ إلى ٢٠٢٦-٠٧-١٣ · العميل: مؤسسة النور"* — ليتأكّد المستخدم قبل الوثوق بالأرقام.
- **عند عدم الفهم:** لا تخمين. رسالة لطيفة + أمثلة قابلة للنقر تملأ الخانة عند النقر.

## عقد JSON

نجاح:
```json
{ "ok": true,
  "understood": "تقرير المبيعات · من 2026-04-14 إلى 2026-07-13 · العميل: مؤسسة النور",
  "kpis": [{"label":"الإجمالي","value":"52,300 ر.س"}],
  "table": {"columns":["الفاتورة","العميل","التاريخ","الإجمالي"],
            "rows":[["INV-1042","مؤسسة النور","2026-07-01","3,200 ر.س"]]},
  "reportUrl": "/sales?from=2026-04-14&to=2026-07-13&q=...",
  "exportUrl": "/export?report=sales&format=csv&from=...&to=...&q=..." }
```
فشل:
```json
{ "ok": false, "message": "لم أفهم الطلب. جرّب أحد الأمثلة:", "suggestions": ["مبيعات آخر ٩٠ يوم","الذمم المتأخرة","الأصناف النافدة"] }
```

- الجدول المصغّر: أعلى ٥–٨ صفوف فقط.
- `reportUrl`/`exportUrl` تُبنى بنفس مفاتيح الفلاتر التي تقبلها الصفحات/الـ`ExportController` الحالية.
- `exportUrl` يظهر فقط للتقارير التي يدعمها `ExportController` (sales, inventory, finance).

## ربط كل تقرير بدواله الموجودة

| report | المصدر | KPIs | صفوف الجدول |
|--------|--------|------|-------------|
| sales | `SalesRepository::summary` + `invoicePage`/`topProducts` | الإجمالي، العدد، المتوسط، العملاء | أحدث الفواتير (أو أعلى المنتجات لو `topN`) |
| customers | `SalesRepository::customerReport` | عدد العملاء، الإجمالي | أعلى العملاء إنفاقًا |
| aging | `SalesRepository::agingInvoices` + `Analytics::agingBuckets` | الإجمالي المستحق، المتأخر | فواتير مستحقة مرتّبة بالأقدم |
| inventory | `InventoryRepository::productPage` + ملخّص المخزون | قيمة المخزون، عدد الأصناف، النافد | أصناف حسب الحالة/البحث |
| finance | `FinanceRepository::transactionPage` + ملخّص | إيرادات، مصروفات، صافي | أحدث الحركات |

> ملاحظة: تُقرأ التواقيع الفعلية للدوال وقت التنفيذ ويُلتزم بها كما هي.

## الأمان

- الاستدعاء GET لنفس النطاق يمرّ عبر `connect-src 'self'` في CSP الحالية — **لا تعديل على CSP**.
- كل المخرجات في JS تُدرَج كنصوص (`textContent`) لا كـ HTML — لا XSS.
- لا SQL جديد؛ كل الاستعلامات عبر الدوال المهيّأة الموجودة (prepared statements + whitelisting).
- `Intent` يطبّع المدخل ويقصّ طوله (حدّ أقصى للحروف) قبل التحليل.

## الاختبارات

- `tests/unit/intent_test.php` — جُمل عربية ← قصد متوقّع: نوع التقرير، تحليل الفترات (نسبي/صريح)، الأرقام العربية، اسم العميل، حالة المخزون، النوع المالي، top-N، نيّة التصدير، وحالات الغموض (report=null). ينضمّ تلقائيًا لـ `tests/run_unit.php`.
- `tests/unit/assistant_test.php` — لكل تقرير: الرد يحوي `ok/understood/kpis` والجدول بالمفاتيح الصحيحة، ويستدعي الدوال الموجودة دون SQL جديد.
- تحقّق E2E للودجت عبر Playwright MCP بعد التنفيذ (فتح، إرسال طلب، ظهور نتيجة)، ولقطة تُحفظ في `tests/screenshots/`.

## خارج النطاق (YAGNI)

- لا حفظ دائم لسجل المحادثة (جلسة المتصفح فقط).
- لا نموذج لغوي ولا مفاتيح ولا اتصال خارجي.
- لا محادثة متعددة الأدوار بسياق تراكمي — كل طلب مستقل.
